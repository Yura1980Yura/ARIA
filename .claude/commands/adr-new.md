Создание новой ADR (Architecture Decision Record).

**Синтаксис:** `/adr-new "{тема}"`

## Что делает

1. Определяет следующий номер ADR (сканирует `docs/ADR/ADR-*.md`)
2. Создаёт файл `docs/ADR/ADR-NNN-{slug}.md` по шаблону
3. Заполняет черновик на основе контекста (SPEC.md, STACK.md, текущей задачи)
4. Предлагает пользователю отредактировать
5. Опционально — добавляет ссылку в SPEC.md

## Шаблон ADR

```markdown
# ADR-NNN: {Тема}

**Статус:** Proposed | Accepted | Deprecated | Superseded by ADR-XXX
**Дата:** YYYY-MM-DD
**Контекст задачи:** {task_name из STATE.yaml если применимо}

## Контекст

{Что происходит, какой выбор стоит перед нами, какие ограничения}

## Рассмотренные варианты

### Вариант A: {название}
- **Плюсы:** ...
- **Минусы:** ...
- **Стоимость внедрения:** ...

### Вариант B: {название}
- **Плюсы:** ...
- **Минусы:** ...
- **Стоимость внедрения:** ...

## Принятое решение

{Какой вариант выбран и почему}

## Последствия

- **Положительные:** ...
- **Отрицательные:** ...
- **Необратимое?** да/нет

## Связанные документы

- SPEC.md раздел X
- STACK.md раздел Y
- ADR-XXX (если зависит)

## Метрики успеха

{Как поймём что решение работает}
```

## Нумерация

- ADR-001, ADR-002, ... — непрерывная нумерация по проекту
- Не переиспользовать номера отклонённых/отменённых ADR
- Deprecated/Superseded ADR остаются в `docs/ADR/` со статусом, не удаляются

## Интеграция с SPEC.md

В SPEC.md обычно есть раздел "Архитектурные решения" или аналог. При принятии ADR — добавить ссылку:

```markdown
## Архитектурные решения

- [ADR-001](ADR/ADR-001-di-hilt-vs-koin.md) — Выбор DI: Hilt vs Koin
- [ADR-002](ADR/ADR-002-map-maplibre.md) — Карта: MapLibre
```

## Правила

- ADR пишется ДО или СРАЗУ после принятия решения, не постфактум через месяц
- Если решение меняется — новый ADR со статусом "Supersedes ADR-NNN", старый обновляется на "Superseded by ADR-MMM"
- ADR коммитится ВМЕСТЕ с кодом, реализующим решение (или вместе с обновлением SPEC.md)
- Scope коммита: `[DOCS]` или проект-специфичный (напр. `[ARCH]`)
