# ARIA

AI-powered development framework for Claude Code. Universal project management system with adapters for any stack.
Bash scripts, Markdown templates, YAML конфигурации.

## ОБЯЗАТЕЛЬНАЯ МОДЕЛЬ: ТОЛЬКО Opus 4.6

**ЗАПРЕЩЕНО использовать Sonnet, Haiku или любую модель кроме Opus 4.6.**

| Модель | Статус |
|--------|--------|
| **Opus 4.6** | ОБЯЗАТЕЛЬНА. Единственная разрешённая модель |
| Sonnet (любая версия) | **ЗАПРЕЩЕНА**. Нарушение = CRITICAL |
| Haiku (любая версия) | **ЗАПРЕЩЕНА**. Нарушение = CRITICAL |

При вызове субагента через Task tool — ВСЕГДА явно указывай `model: "opus"`.

## Переменные окружения (upstream self-contained после v3.4 миграции 2026-04-15)

| Переменная | Назначение |
|---|---|
| ARIA_DOCS | Корень проекта (Yandex.Disk/Obsidian/W4_ARIA) — и docs, и код, и команды, и git remote |
| ARIA_CODE | = ARIA_DOCS (оставлено для совместимости с шаблонами команд, использующими `{{ENV_CODE}}`) |

**Правило разделения репо (правило 4 Task 11) применяется к форкам с application code**, не к upstream ARIA. Upstream — признанное исключение: ARIA является и «кодом» (команды/шаблоны/адаптеры) и «доками» (SPEC/STACK/ADR/policies) одновременно, всё в одном каталоге.

## Конфигурация путей

**Все пути к документам, спекам, тестам — в `$ARIA_DOCS/PATHS.yaml`.**

## При входе в чат

1. Прочитай STATE.yaml — ТОЛЬКО секцию текущей фазы + current + blockers
2. Прочитай spec задачи: {specs}/{task_name}.md (если есть)
3. НЕ читай CHANGELOG и ROADMAP — они обновляются автоматически

## Документооборот (ARIA v3.3)

### Принцип: один документ — одна функция — замкнутый контур

| Документ | Назначение | Кто создаёт | Кто читает |
|----------|-----------|-------------|-----------|
| docs/spec/{task}.md | Полная спецификация задачи | /spec | Кодер, Verify, Code Review |
| docs/SPEC.md | Глобальная архитектура ARIA | /spec (раздел) | Все |
| docs/ADR/*.md | Архитектурные решения | /spec (автогенерация) | Аналитик, Code Review |
| docs/REFERENCES.md | Индекс референсов | /spec Research | Аналитик |
| docs/research/{topic}.md | Свободные исследования | /research | Аналитик |
| STATE.yaml | Состояние проекта | /done, /spec | Все |

### read_docs — связующий механизм:
spec/{task}.md содержит YAML-заголовок с `read_docs` и `read_code`.
Оркестратор при кодировании читает read_docs из spec.

## Как решать проблемы (ОБЯЗАТЕЛЬНЫЙ ПРОТОКОЛ)

**ГЛАВНОЕ ПРАВИЛО: НЕ ЧИНИ СИМПТОМЫ. ЧИНИ ПРИЧИНЫ.**

### 1. СТОП
Не пиши ни строки кода. Сначала пойми что происходит.

### 2. МАСШТАБ

| Уровень | Признаки | Действие |
|---------|----------|----------|
| **Локальный** | Опечатка, пропущенный аргумент | Чини сразу |
| **Контрактный** | Шаблон не совпадает с ожиданиями команды | → Шаг 3 |
| **Архитектурный** | Текущее решение не способно поддержать нужное поведение | → Шаг 3 + эскалация |

### 3. АУДИТ

Прочитай ВСЕ связанные файлы. Выведи:
```
ПРОБЛЕМА: [что сломалось]
ЗАТРОНУТЫ: [файлы]
ПРИЧИНА: [почему сломалось — не симптом, а корень]
РЕШЕНИЕ: [план]
```

### 4. КОРНЕВАЯ ПРИЧИНА (3 раза «почему?»)
Чинить нужно последний «почему», а не первый.

### 5. ПЛАН → КОД → ВСЕ ТЕСТЫ
Опиши решение до написания кода. После — запусти ВСЕ тесты.

---

### ЗАПРЕТ НА УГАДЫВАНИЕ (ПРАВИЛО ОДНОЙ ГИПОТЕЗЫ)

**Первая попытка не решила проблему — ПРЕКРАТИ ПИСАТЬ КОД.**

Вместо второй попытки — ОБЯЗАТЕЛЬНО одно из:
**A) Реальные данные.** Логи, значения переменных, состояние объектов.
**B) Minimal reproduction.** Новый файл, 50 строк, только баг.
**C) Документация.** Открой docs, найди раздел, прочитай.
**D) Эскалация.** Blocker в STATE.yaml.

### ПРАВИЛО ОДНОЙ ПРОБЛЕМЫ
**Решай ОДНУ проблему за раз.**

### ПРАВИЛО "НЕ ТРОГАЙ ТО ЧТО РАБОТАЕТ"
Перед каждым изменением: "Это точно нужно менять?"

### ПРАВИЛО СТАНДАРТНОГО РЕШЕНИЯ
**Стандартное решение ВСЕГДА лучше кастомного.**

---

### ЗАПРЕЩЁННЫЕ ПАТТЕРНЫ

| Костыль | Почему запрещён |
|---------|----------------|
| Подавление предупреждений | Прячет ошибку |
| `# TODO: fix later` / `# HACK` | «Потом» не наступит |
| Копирование логики из другого файла | Копипаста вместо импорта |
| Hardcoded пути | Все пути в PATHS.yaml |

### КРИТИЧЕСКИЕ АНТИПАТТЕРНЫ ПРОЕКТИРОВАНИЯ

| Антипаттерн | Почему запрещён |
|-------------|----------------|
| MVP-мышление ("потом доделаем") | Потом не наступит. Каждое решение = production-grade |
| Костыли и workaround-ы | Костыль привлекает костыли |
| Hardcoded параметры | Все параметры — в конфигурации/шаблонах |

### ПРАВИЛО ТРЁХ ФАЙЛОВ
Если «локальный фикс» требует изменений в 3+ файлах — СТОП.

### ПРАВИЛО ТРЁХ ПОПЫТОК
Первая → данные. Вторая → откат. Третья → эскалация (blocker).

## После завершения задачи
Выполни /done — тесты → коммит → push → STATE → CHANGELOG

## Формат коммитов

[CORE] описание — Шаблоны, команды, протоколы
[PROTOCOL] описание — Протоколы синхронизации
[ADAPTER] описание — Адаптеры (python-fastapi, kotlin-android, csharp-avalonia)
[POLICY] описание — Политики (changelog, commit, documentation)
[DOCS] описание — Документация (ARIA_GUIDE, README, SPEC, STACK, policies)
[HOOKS] описание — Git hooks (pre-commit, commit-msg, pre-push)
[FIX] описание — Исправления
[META] описание — Версионирование, конфигурация

## Git routing (upstream self-contained после v3.4)

Один репозиторий (`W4_ARIA`) — git init внутри Yandex.Disk-папки, remote = `https://github.com/Yura1980Yura/ARIA.git`.

Обычный коммит (Two-Phase Commit не применим для upstream — нет разделения code/docs):
```bash
cd "$ARIA_DOCS"
git add <файлы>
git commit -m "[SCOPE] описание"
git push origin main
```

**Для форков** (с application code) Two-Phase Commit остаётся обязательным:
- Phase 1 — код приложения в code-repo
- Phase 2 — docs через junction `project-docs/` → docs-repo в облачном хранилище

## Правила документов (ARIA Phase 1, Task 11)

Полные правила — в `core/CLAUDE.md.template` (тот же файл подставляется в форки). Краткий список:

1. **Живой документ:** AI пишет, AI или hook читает. Пользователь не редактирует вручную (исключения: README/LICENSE, ROADMAP через hook, policies только через чат).
2. **Необходимость документа:** тест абсентии перед созданием. Срезы/обзоры запрещены → команды.
3. **Policies через чат:** `policies/*`, `CLAUDE.md`, `CHANGELOG_POLICY`, `DOCUMENTATION_LIFECYCLE` редактируются только через диалог с AI.
4. **Разделение репо:** для форков с application code — ARIA-инфраструктура только в docs-repo. Upstream ARIA — исключение (self-contained).
5. **Контракт чтения:** каждый документ имеет писателя и читателя (команду/hook).
