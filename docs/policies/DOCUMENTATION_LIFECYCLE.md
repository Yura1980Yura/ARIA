# Documentation Lifecycle

Какой документ обновляется при каком событии. Применяется **семантически** через CLAUDE.md и **механически** через git hooks.

---

## Матрица обновлений

### В upstream ARIA

| Событие | Обязательно обновить | Рекомендовано |
|---------|---------------------|---------------|
| Новая команда в `core/commands/` | `CHANGELOG.md`, `docs/ARIA_GUIDE.md` (список команд) | — |
| Изменение команды | `CHANGELOG.md` | `docs/ARIA_GUIDE.md` если семантика |
| Новый `core/protocols/*.md` | `CHANGELOG.md`, `docs/ARIA_GUIDE.md` | — |
| Изменение `core/CLAUDE.md.template` | `CHANGELOG.md` | `docs/ARIA_GUIDE.md` |
| Новый адаптер | `CHANGELOG.md`, `docs/ARIA_GUIDE.md` (список адаптеров) | — |
| Изменение адаптера | `CHANGELOG.md` | — |
| Новая политика в `docs/policies/` | `CHANGELOG.md`, `docs/ARIA_GUIDE.md` (список политик) | — |
| Изменение hook-скрипта | `CHANGELOG.md` | `docs/policies/*` если поменялись правила |
| Принята контрибуция из форка (`/aria-triage --accept`) | `CHANGELOG.md` (строка таблицы с колонкой `атрибуция: from {fork}, PR #{N}`), `TRIAGE.md` (→ архив), `FORKS.md` (если новый Repo) | — |
| Релиз новой версии | `CHANGELOG.md` с версией + тег | — |

| Изменение структуры модулей | `docs/SPEC.md` | `ADR-NNN` |

## Принцип "живой документ" (ARIA Phase 1, Task 11)

Документ **жив** если:
1. **Его пишет AI или git-hook** (не пользователь вручную). Записи создаются в рамках команд `/done`, `/spec`, `/aria-sync`, `/aria-triage`, `/aria-release` или автогенерацией.
2. **Его читает механически AI или другой автомат** в рамках workflow (не только глазами пользователя).

Документ **мёртв** если:
1. Никто его не читает в рамках автоматизации → превращается в legacy.
2. Никто не проверяет → расходится с реальностью.
3. **Пользователь редактирует его вручную** → теряется аудит-trail, нарушается целостность.

**ПРАВИЛО: пользователь общается с системой ТОЛЬКО через чат с AI.** Любое ручное редактирование файлов проекта (кроме кода в целевом репо) — нарушение целостности.

### Исключения (признанные классы)

1. **Документы для внешней аудитории** (`README.md`, `LICENSE`) — AI пишет при триггер-событиях, внешние посетители читают. Пользователь upstream-а не редактирует.
2. **Документы с валидатором-hook'ом** (`ROADMAP.md`) — pre-commit валидатор считается механическим читателем уровня CI. Применяется ТОЛЬКО когда hook реально существует.
3. **Policies** (`docs/policies/*.md`, `CLAUDE.md`, `CHANGELOG_POLICY.md`, `DOCUMENTATION_LIFECYCLE.md`, `COMMIT_POLICY.md`) — обновляются **только через чат с AI**. Редактор пользователя запрещён. Механически не валидируются, но защищены правилом «policies через чат».

### Правило необходимости документа

Перед созданием любого документа — тест абсентии:
- «Если не создавать — что сломается?»
- «Можно ли получить ту же информацию из других документов / git / автогенерации?»
- «Сколько раз в год кто-то его откроет?»

Документ создаётся только если он **единственный** источник данных/истины для конкретного потребителя. Срезы, обзоры, «для полноты картины» — **запрещены**, заменяются командами, генерирующими срез по запросу.

### Правило разделения репозиториев

ARIA-инфраструктура (`.claude/`, `CLAUDE.md`, `STATE.yaml`, `SPEC.md`, `STACK.md`, `REFERENCES.md`, `CHANGELOG.md`, `ROADMAP.md`, `SYS_CHANGELOG.md`, `PATHS.yaml`, `project_config.yaml`, `docs/`, `reports/`, `backlog/`) живёт **ТОЛЬКО в docs-репо**. Code-репо содержит **ТОЛЬКО код приложения**.

Исключение: `.mcp.json` — стандарт Claude Code, может быть в обоих репо (но контент идентичен).

Обмен между репо — через Two-Phase Commit (junction `project-docs/`), **никогда через rsync**.

Нарушения блокируются:
- `/auto` шаг 0.2 — проверка чистоты code-репо
- `/aria-docs-audit` — аудит обоих репо
- `.gitignore` в code-репо — запрет коммита ARIA-артефактов

## Валидация документа на "живость"

`/aria-docs-audit` в upstream проверяет:

### 1. Синхронизация контента
1. Соответствует ли список команд в `docs/ARIA_GUIDE.md` реальному содержимому `core/commands/`?
2. Соответствует ли список адаптеров в `docs/ARIA_GUIDE.md` реальному `adapters/`?
3. Соответствует ли список политик?
4. Все ли протоколы упомянуты в ARIA_GUIDE?

### 2. Живость документа (каждый .md в docs/)
Для каждого документа:
- Есть ли писатель-автомат (grep по команде/hook, который обновляет файл)?
- Есть ли читатель-автомат (grep по команде/hook, который читает файл)?
- Если нет — флагнуть как DEAD, предложить: реализовать контур / удалить / переклассифицировать.

### 3. Разделение репозиториев
В code-repo ($ENV_CODE) НЕ должно быть:
`.claude/`, `.dev/`, `CLAUDE.md`, `STATE.yaml`, `CHANGELOG.md`, `ROADMAP.md`, `SYS_CHANGELOG.md`, `SPEC.md`, `STACK.md`, `REFERENCES.md`, `PATHS.yaml`, `project_config.yaml`, `docs/`, `reports/`, `backlog/`.
При нарушении — FAIL с перечнем найденных путей.

### 4. Синхронизация .mcp.json
Если `.mcp.json` есть в обоих репо — md5 должны совпадать. Расхождение — WARNING.

### 5. Политики vs hooks
Hook `pre-commit` проверяет:
1. Если изменён `core/commands/X.md` — затронут ли `ARIA_GUIDE.md`?
2. Если добавлен новый адаптер — обновлён ли `ARIA_GUIDE.md`?
3. Если затронуты `core/`, `adapters/`, `docs/policies/` — обновлён ли `CHANGELOG.md`?

## Семантический контур

`core/CLAUDE.md.template` ссылается на эту политику. AI-агент при любом изменении в ARIA обязан:
1. Прочитать этот файл
2. Определить событие из матрицы выше
3. Перед коммитом обновить все "обязательно" документы
4. Упомянуть "рекомендованные" в обсуждении с пользователем

---

## Механический контур

Git hooks (`scripts/hooks/pre-commit`, `pre-push`) читают эти же правила (закодированы в скрипте) и блокируют коммит при нарушении. Это гарантирует что политика не остаётся "бумажной".

**Критически важно:** если политика изменена (эта таблица) — должен быть изменён и hook-скрипт. Hook-тесты (`scripts/hooks/tests/`) проверяют соответствие политики и hook-кода.
