## Цель

Довести ARIA до состояния, когда новый пользователь может:
1. Развернуть ARIA под свой проект одной командой (включая генеральную спеку продукта)
2. Работать с полным конвейером (spec → code → review → done)
3. Синхронизироваться с upstream
4. Отправлять contribute-back в правильном формате

А владелец upstream может:
1. Принимать и обрабатывать предложения от форков (пакетно)
2. Выпускать релизы с полной прослеживаемостью
3. Уведомлять форки об обновлениях автоматически из FORKS.md

Все документы ARIA должны соответствовать правилам живого документа и необходимости документа.

## Принципы Phase 1

- **Замкнутые контуры.** Каждый новый документ/команда имеет семантического + механического потребителя, включая писателя
- **Правило живого документа.** Пользователь не редактирует документы проекта вручную — только через чат с AI
- **Правило необходимости документа.** Не бывает документов ради документов. Документ создаётся тогда и только тогда, когда он единственный источник данных/истины для конкретного потребителя
- **Git + gh CLI = инфраструктура.** Никакого кастомного ПО
- **Стандартные решения.** `.mcp.json` (стандарт Claude Code), GitHub PRs (стандарт git), YAML/Markdown (стандарт ARIA)
- **Production-grade.** Каждая задача — полностью завершённая, не заглушка
- **Запрет откладывать.** Всё в Phase 1. Ничего в «Phase 2 / потом»

### Task 11: Правила живого документа и необходимости документа
**Приоритет:** P1
**Scope:** `[DOCS] + [POLICY]`
**Зависимости:** нет — делается первой, блокирует все остальные

**Описание:**
Зафиксировать в CLAUDE.md и DOCUMENTATION_LIFECYCLE.md два мета-правила, провести аудит всех существующих документов ARIA на соответствие.

**Правило живого документа:**
> Документ жив тогда и только тогда, когда:
> 1. Пишет его AI или git hook (не пользователь вручную)
> 2. Читает его механически AI или другой автомат (не только глазами пользователя)
>
> Исключение: документы для внешней аудитории (README, LICENSE) — AI пишет, внешние люди читают. Пользователь upstream-а их всё равно не трогает.
>
> Пользователь общается с системой ТОЛЬКО через чат. Любое ручное редактирование файлов проекта (кроме кода в целевом репо) — нарушение целостности.

**Правило необходимости документа:**
> Перед созданием любого документа — тест абсентии:
> 1. «Если этот документ не создавать — что сломается?»
> 2. «Можно ли получить ту же информацию из других документов / git / автогенерации?»
> 3. «Сколько раз в год кто-то (AI или человек) его откроет?»
>
> Документ создаётся тогда и только тогда, когда:
> - Он ЕДИНСТВЕННЫЙ источник данных для автоматизации ИЛИ
> - Он ЕДИНСТВЕННЫЙ источник истины для человека при принятии решений
> - Кто-то (AI или человек) действительно его читает регулярно
>
> Документация — дорогой артефакт. Срезы, обзоры, «для полноты картины» — запрещены. Вместо них — команды, генерирующие срез по запросу.
