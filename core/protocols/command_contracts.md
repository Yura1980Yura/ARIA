# ARIA — Контракты минимальной функциональности команд

> **Писатель:** upstream-maintainer (через чат с AI, правило 3 policies).
> **Читатели:** `/aria-docs-audit` (проверка custom-команд форка на минимум), `/aria-sync` при pull (предупреждение если custom-версия ниже минимума).
> Этот документ определяет ОБЯЗАТЕЛЬНЫЙ минимум для каждой fork-level команды. Форк может кастомизировать (INTENTIONALLY_CUSTOM), но не ниже контракта.

---

## Зачем

Форк с `intentionally_custom` команда может убрать критичные этапы (E2E Testing Plan, Adversarial Review, REFERENCES контур) и потерять production-grade гарантии без единого предупреждения. Этот контракт определяет минимальный набор, который ОБЯЗАН сохраняться даже в custom-версии.

---

## Формат контракта

Каждая команда имеет список **обязательных элементов**. Элемент = секция/шаг/механизм, без которого команда теряет production-grade гарантии.

Проверка: текстовый поиск (grep) по содержимому команды в форке. Если обязательный элемент не найден — WARN при `/aria-docs-audit`, WARN при `/aria-sync pull`.

---

## Контракты fork-level команд

### /spec

| # | Обязательный элемент | Grep-паттерн | Почему обязателен |
|---|---------------------|-------------|-------------------|
| 1 | E2E Testing Plan секция | `E2E Testing Plan` | Без E2E плана тесты будут случайными |
| 2 | Test Coverage Matrix (TCM) | `Test Coverage Matrix\|TCM` | Матрица покрытия — основа полноты тестов |
| 3 | Acceptance Criteria | `acceptance\|Acceptance\|AC` | Без AC нет формального критерия done |
| 4 | Риски секция | `[Рр]иск` | Без оценки рисков — слепая разработка |
| 5 | read_docs в YAML-заголовке | `read_docs` | Связь spec с документацией |

### /review

| # | Обязательный элемент | Grep-паттерн | Почему обязателен |
|---|---------------------|-------------|-------------------|
| 1 | Минимум 3 ревьюера | `Субагент [1-3]\|ревьюер` | 1-2 ревьюера = слепые зоны |
| 2 | Verify (build+spec) | `[Vv]erify\|build.*spec\|spec.*compliance` | Без verify — ревью без фактов |
| 3 | Code Review | `[Cc]ode [Rr]eview\|code review` | Качество кода |
| 4 | Adversarial Tester | `[Aa]dversarial` | Без атакера — happy path only |
| 5 | VERDICT формат | `VERDICT.*PASS\|NEEDS_FIX` | Без формального вердикта — ревью бесполезно |

### /auto

| # | Обязательный элемент | Grep-паттерн | Почему обязателен |
|---|---------------------|-------------|-------------------|
| 1 | Инфра-проверка | `[Ии]нфра.*проверка\|INFRA_CHECK` | Без инфра-чека — сломанные сборки |
| 2 | Spec чтение | `spec.*задачи\|spec/{task` | Без spec — кодирование наугад |
| 3 | Quality Gate | `Quality Gate\|QUALITY.*GATE` | Без gate — пропуск багов |
| 4 | /review вызов | `/review\|РЕВЬЮ` | Без ревью — code goes to prod unchecked |
| 5 | E2E Verify | `E2E.*[Vv]erify\|e2e-gate` | Без E2E — интеграционные баги |
| 6 | /done вызов | `/done\|коммит.*push` | Без done — незафиксированная работа |
| 7 | Лимит попыток | `макс.*3\|максимум.*3\|3 попыт` | Без лимита — бесконечный цикл |

### /done

| # | Обязательный элемент | Grep-паттерн | Почему обязателен |

| 1 | Phase 1 — Code Commit | `Phase 1\|[Cc]ode [Cc]ommit` | Разделение code/docs |

| 3 | STATE.yaml обновление | `STATE.yaml` | Прогресс зафиксирован |
| 4 | CHANGELOG обновление | `CHANGELOG` | История изменений |

| # | Обязательный элемент | Grep-паттерн | Почему обязателен |
|---|---------------------|-------------|-------------------|

| 2 | Spec чтение | `spec.*задачи\|spec/{task` | Понимание задачи |

### /aria-sync

| # | Обязательный элемент | Grep-паттерн | Почему обязателен |

### /status, /adr-new, /aria-docs-audit, /e2e-gate, /roadmap-sync

Контракт: минимальный (команда должна существовать и быть непустой, >10 строк).

## Как используется

### `/aria-docs-audit` (проверка 7 — новая)

Для каждой fork-level команды в `.claude/commands/`:
1. Прочитать содержимое
2. Для каждого обязательного элемента из контракта — grep
3. Если элемент не найден — WARN: `{command} не содержит обязательный элемент: {name}`
4. Если >50% элементов отсутствуют — ERROR: `{command} ниже минимального контракта`

### `/aria-sync` при pull (дополнение к INTENTIONALLY_CUSTOM)

Если артефакт помечен как INTENTIONALLY_CUSTOM:
1. Проверить контракт минимальной функциональности
2. Если custom-версия ниже минимума — WARN пользователю:
   ```
   WARN: {command}.md помечен как INTENTIONALLY_CUSTOM, но ниже минимального контракта.
   Отсутствуют: {список элементов}
   Рекомендация: добавить недостающие элементы или принять upstream-версию.
   ```

---

## Обновление контракта

Контракт обновляется через чат с AI (правило 3 policies). При добавлении нового обязательного элемента — все форки увидят WARN при следующем `/aria-docs-audit` или `/aria-sync`.
